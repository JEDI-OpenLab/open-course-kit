Structurer et publier

Utiliser GitHub pour publier une ressource pédagogique ouverte

GitHub peut servir à publier bien plus que du code. Pour une ressource pédagogique ouverte, il permet de conserver les fichiers sources, documenter les usages, suivre les évolutions, afficher une licence claire et publier facilement un mini-site avec GitHub Pages.

Cette page propose une structure simple pour transformer un dossier pédagogique en dépôt clair, réutilisable et publiable.

Point de vigilance

GitHub est un outil puissant, mais il peut être intimidant pour des publics non techniques. Une bonne publication pédagogique doit donc rester lisible sans supposer que les utilisateurs connaissent Git, les branches, les commits ou les pull requests.

Pourquoi GitHub pour une ressource pédagogique ?

Centraliser

Regrouper au même endroit les supports, fichiers sources, images, modèles, licences, crédits et documentations associées.

Documenter

Donner un contexte d’usage, expliquer comment reprendre la ressource, indiquer la licence et signaler les contenus tiers.

Publier

Transformer un dépôt en site web public grâce à GitHub Pages, sans infrastructure lourde ni serveur à maintenir.

Structure minimale recommandée

Un dépôt utile n’a pas besoin d’être complexe. Pour commencer, il doit surtout contenir un fichier de présentation, une licence, une documentation et les fichiers réutilisables.

Arborescence minimale

ma-ressource-ouverte/
├── README.md
├── LICENSE
├── docs/
│   └── index.html
└── assets/
    ├── images/
    └── fichiers/
Rôle des principaux fichiers
Élément Rôle Priorité
README.md Présente le projet, son public, ses usages, sa licence et ses conditions de réutilisation. Indispensable
LICENSE Indique la licence applicable aux contenus originaux ou au code selon le cas. Indispensable
docs/ Contient les pages publiées avec GitHub Pages ou la documentation principale. Fortement conseillé
assets/ Regroupe les images, fichiers téléchargeables, médias ou documents associés. Conseillé

Structure plus complète

Pour une ressource plus ambitieuse, il peut être utile de séparer les sources modifiables, les crédits, les exemples, les exports et la documentation publiée.

Arborescence conseillée

ma-ressource-ouverte/

├── LICENSE
├── CITATION.cff
├── CHANGELOG.md
├── docs/
│   ├── index.html
│   ├── guide.html
│   └── ressources.html
├── assets/
│   ├── images/

│   ├── documents-modifiables/
│   ├── donnees/

Le README : la porte d’entrée du dépôt

Le fichier README.md est souvent le premier élément lu dans un dépôt GitHub. Il doit permettre de comprendre rapidement ce qu’est la ressource, à qui elle s’adresse, comment l’utiliser et dans quelles conditions elle peut être reprise.

Un README doit dire

  • ce qu’est le projet ;
  • à qui il s’adresse ;
  • ce qu’il contient ;
  • comment l’utiliser ;
  • comment le citer ;
  • sous quelle licence il est publié.

Un README doit éviter

  • les descriptions trop vagues ;
  • les fichiers non expliqués ;
  • les licences absentes ;
  • les droits tiers non signalés ;
  • les consignes compréhensibles uniquement par l’auteur.

Modèle court de README

# Titre de la ressource

## Présentation

Courte description de la ressource, de son objectif et de son contexte de production.

## Public cible

Indiquer les publics concernés : enseignants, étudiants, formateurs, ingénieurs pédagogiques, etc.

## Objectifs pédagogiques

- Objectif 1
- Objectif 2
- Objectif 3

## Contenu du dépôt

- `docs/` : documentation ou site publié
- `assets/` : images et fichiers associés
- `sources/` : fichiers modifiables ou sources
- `credits/` : sources, crédits et droits tiers

## Utilisation

Expliquer comment consulter, adapter ou utiliser la ressource.

## Licence

Ressource éducative libre (REL) au sens de la Recommandation UNESCO de 2019. Sauf mention contraire, les contenus originaux sont placés sous licence Creative Commons Attribution 4.0 International — CC BY 4.0.

Les contenus tiers conservent leurs droits et licences propres. Voir le site principal de JEDI-OpenLab pour plus d’informations.

## Citation

Indiquer comment citer la ressource.

## Contact

Indiquer un contact, un lien vers le projet ou une page de contribution.

Publier avec GitHub Pages

GitHub Pages permet de publier un site web statique à partir d’un dépôt. Pour Open Course Kit, c’est une solution adaptée aux guides, documentations, petits outils HTML/CSS/JavaScript, ressources pédagogiques et pages de valorisation.

1

Créer le dépôt

Créez un dépôt GitHub avec un nom explicite, court et lisible. Ajoutez une description claire et, si possible, quelques mots-clés.

2

Ajouter les fichiers

Ajoutez au minimum un README.md, un fichier LICENSE, un dossier docs/ et une page index.html dans ce dossier.

3

Activer GitHub Pages

Dans les paramètres du dépôt, activez GitHub Pages en choisissant la branche principale et le dossier de publication, par exemple /docs ou la racine du dépôt.

4

Tester le site

Vérifiez l’URL publique, testez les liens, les images, les fichiers téléchargeables, l’affichage mobile et la lisibilité des pages principales.

Nommer les fichiers simplement

Des noms de fichiers clairs rendent un dépôt beaucoup plus facile à parcourir. Mieux vaut éviter les noms trop longs, les espaces, les accents et les versions ambiguës comme final-v3-vrai-final.docx. Un nom stable et descriptif facilite la reprise par d’autres utilisateurs.

Exemples de nommage
À éviter Préférer Pourquoi
Mon cours final version 2.docx guide-animation.md Lisible, stable, sans espace ni ambiguïté.
Image écran 23 mai.png capture-interface.png Décrit le contenu plutôt que la date de capture.
nouveau truc test.html prototype-activite.html Indique la fonction du fichier.
sources diverses.md sources-et-droits.md Clarifie le rôle juridique et documentaire du fichier.

Ajouter une citation avec CITATION.cff

Pour faciliter la citation d’une ressource, il est possible d’ajouter un fichier CITATION.cff à la racine du dépôt. GitHub le reconnaît et peut afficher un bouton “Cite this repository”. C’est utile pour les ressources pédagogiques, les guides, les corpus, les jeux de données ou les démonstrateurs.

Modèle simple de CITATION.cff

cff-version: 1.2.0
title: "Titre de la ressource"
message: "Si vous utilisez cette ressource, merci de la citer."
type: educational-resource
authors:
  - family-names: "Nom"
    given-names: "Prénom"
year: 2026
license: CC-BY-4.0
url: "https://exemple.github.io/ma-ressource/"

Documenter les évolutions avec un changelog

Un fichier CHANGELOG.md permet d’indiquer les principales évolutions d’une ressource : ajout d’un chapitre, correction d’une erreur, changement de licence, nouvelle version d’un fichier ou amélioration d’un modèle.

Modèle court de CHANGELOG

# Changelog

## 1.0.0 — 2026-05-23

### Ajouté

- Première version publique de la ressource.
- Ajout du README, de la licence et de la documentation.

### Modifié

- Aucun changement pour cette première version.

### Corrigé

- Aucun correctif pour cette première version.

Clarifier les droits dans le dépôt

Le dépôt doit éviter toute ambiguïté : ce qui est original peut être placé sous une licence ouverte, mais les contenus tiers doivent rester associés à leurs propres conditions d’usage.

Checklist GitHub avant publication

Cette checklist peut être utilisée avant de rendre le dépôt public ou avant d’activer GitHub Pages.

Modèle copiable

## Checklist GitHub

- [ ] Le nom du dépôt est clair et lisible.
- [ ] La description du dépôt indique l’objectif de la ressource.
- [ ] Le dépôt contient un README.md complet.
- [ ] Le dépôt contient un fichier LICENSE.
- [ ] La licence est aussi mentionnée dans le README.
- [ ] Les contenus tiers sont signalés dans une section ou un fichier dédié.
- [ ] Les fichiers sont organisés dans des dossiers compréhensibles.
- [ ] Les fichiers importants ont des noms simples, sans espaces ni accents.
- [ ] Les formats sources modifiables sont fournis quand c’est pertinent.
- [ ] Les liens internes fonctionnent.
- [ ] Les images s’affichent correctement.
- [ ] GitHub Pages est activé si un site public est prévu.
- [ ] L’URL publique a été testée.
- [ ] Le dépôt ne contient pas de données personnelles ou confidentielles.

Erreurs fréquentes à éviter

Un dépôt sans README

Sans README, une ressource peut être techniquement disponible mais pratiquement inutilisable. Le README est le mode d’emploi minimal.

Une licence absente ou ambiguë

Sans licence claire, les visiteurs ne savent pas ce qu’ils peuvent faire avec la ressource. Par prudence, ils risquent de ne pas la réutiliser.

Des fichiers sources oubliés

Publier seulement une version finale peut limiter les adaptations. Les fichiers modifiables augmentent fortement la valeur d’usage.

Un site publié mais non testé

Liens cassés, images absentes, chemins incorrects : mieux vaut tester le site comme si l’on découvrait la ressource pour la première fois.

À retenir

Un bon dépôt GitHub pédagogique n’est pas seulement un espace de stockage. C’est une ressource documentée, citable, lisible, réutilisable et publiable. Le triptyque minimal : README.md, LICENSE, documentation claire. Le reste peut venir ensuite.
