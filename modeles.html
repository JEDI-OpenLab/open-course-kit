Modèles et checklists

Des blocs prêts à copier pour publier une ressource ouverte

Cette page rassemble les modèles utiles pour préparer, documenter et publier une ressource pédagogique ouverte : README, licence, crédits, citation, fiche ressource, checklists et arborescences.

Les modèles sont volontairement génériques. Ils sont faits pour être copiés, adaptés, raccourcis, complétés ou intégrés dans un dépôt GitHub, un site de cours, une documentation ou un dossier projet.

Mode d’emploi

Remplacez les passages entre crochets par vos propres informations. Supprimez les sections inutiles. Ajoutez les précisions nécessaires sur les droits tiers, les sources, les contributeurs et les limites d’usage. Une bonne documentation est une documentation que quelqu’un d’autre peut comprendre sans connaître son contexte initial.

Modèle de README.md

Le README est la porte d’entrée du dépôt. Il doit expliquer rapidement ce que contient la ressource, à qui elle s’adresse, comment l’utiliser et sous quelle licence elle est publiée.

## Présentation

[Décrire brièvement la ressource, son objectif, son contexte de production et son intérêt pédagogique.]

## Public cible

Cette ressource s’adresse à :

- [public 1]
- [public 2]
- [public 3]

## Objectifs pédagogiques

À l’issue de l’utilisation de cette ressource, les participants pourront :

- [objectif 1]
- [objectif 2]
- [objectif 3]

## Contenu du dépôt

- `docs/` : documentation ou site publié
- `assets/` : images, médias et fichiers associés
- `sources/` : fichiers sources modifiables
- `credits/` : sources, crédits et droits tiers

Modèle de section licence CC0

Ce modèle convient aux ressources pédagogiques originales que l’on souhaite rendre aussi réutilisables que possible. Il peut être placé dans un README, une page “À propos” ou un fichier dédié. 

## Licence

Sauf mention contraire, les contenus, documents, exemples, fichiers web et ressources originales du projet sont placés sous [CC0 1.0 Universal](https://creativecommons.org/publicdomain/zero/1.0/deed.fr).

Cela signifie que l’auteur renonce, dans toute la mesure permise par la loi, à ses droits patrimoniaux sur ces contenus afin qu’ils puissent être copiés, adaptés, réutilisés, republiés et transformés sans demander d’autorisation.

Les marques, logiciels, services tiers, documents sources importés dans le corpus et ressources citées restent soumis à leurs licences et droits respectifs.

Modèle de double licence contenus / code

Si le projet contient à la fois des contenus pédagogiques et du code informatique, il peut être utile de distinguer les deux. 

## Licence

Sauf mention contraire, les contenus pédagogiques originaux de ce projet sont placés sous [CC0 1.0 Universal](https://creativecommons.org/publicdomain/zero/1.0/deed.fr).

Le code source du projet est placé sous licence [MIT / Apache 2.0 / GPL / autre licence à préciser].

Les marques, logiciels, services tiers, bibliothèques, documents sources importés et ressources citées restent soumis à leurs licences et droits respectifs.

Modèles de citation

Une ressource ouverte gagne à proposer une forme de citation simple. Cela facilite la réutilisation et permet de garder une trace de l’origine du travail, même lorsqu’une licence très permissive comme CC0 est choisie. 

## Comment citer cette ressource ?

Format recommandé :

> [Nom, Prénom]. [Titre de la ressource]. [Nom du projet ou du site]. [Année]. [URL].

Exemple :

> Debaecker, Jean. Open Course Kit. JEDI-OpenLab. 2026. [URL du projet].

Modèle CITATION.cff

cff-version: 1.2.0
title: "[Titre de la ressource]"
message: "Si vous utilisez cette ressource, vous pouvez la citer ainsi."
type: educational-resource
authors:
  - family-names: "[Nom]"
    given-names: "[Prénom]"
year: 2026
license: CC0-1.0
url: "[URL de la ressource]"

Modèle de page d’accueil HTML

Ce modèle minimal peut servir de point de départ pour une page publiée avec GitHub Pages. 

<!DOCTYPE html>
<html lang="fr">
<head>
  <meta charset="UTF-8" />
  <meta name="viewport" content="width=device-width, initial-scale=1.0" />
  <title>[Titre de la ressource]</title>
  <meta name="description" content="[Description courte de la ressource.]" />
</head>
<body>
  <header>
    <h1>[Titre de la ressource]</h1>
    <p>[Phrase de présentation.]</p>
  </header>

  <main>
    <section>
      <h2>Présentation</h2>
      <p>[Décrire la ressource, son public et son usage.]</p>
    </section>

    <section>
      <h2>Utilisation</h2>
      <p>[Expliquer comment consulter ou réutiliser la ressource.]</p>
    </section>

    <section>
      <h2>Licence</h2>
      <p>Sauf mention contraire, les contenus originaux sont placés sous CC0 1.0 Universal.</p>
    </section>
  </main>
</body>
</html>

Checklists copiables

Checklist avant publication

## Checklist avant publication

- [ ] Les auteurs et contributeurs sont identifiés.
- [ ] Les contenus originaux sont distingués des contenus tiers.
- [ ] La licence des contenus originaux est indiquée.
- [ ] Les images, icônes, captures d’écran et médias ont été vérifiés.
- [ ] Les extraits textuels et citations sont correctement attribués.
- [ ] Les données personnelles ont été retirées ou anonymisées.
- [ ] Les documents internes ou confidentiels ont été retirés.
- [ ] Les marques, logos et services tiers sont signalés comme tels.
- [ ] Une section “Crédits et sources” est présente.
- [ ] Un fichier LICENSE est présent.
- [ ] Un README.md décrit le projet et ses conditions de réutilisation.
- [ ] Les fichiers sont organisés dans une arborescence compréhensible.
- [ ] Les formats sources modifiables sont fournis quand c’est pertinent.
- [ ] Les consignes sont compréhensibles hors contexte.
- [ ] Les liens principaux ont été testés.

Checklist GitHub

## Checklist GitHub

- [ ] Le nom du dépôt est clair et lisible.
- [ ] La description du dépôt indique l’objectif de la ressource.
- [ ] Le dépôt contient un README.md complet.
- [ ] Le dépôt contient un fichier LICENSE.
- [ ] La licence est aussi mentionnée dans le README.
- [ ] Les contenus tiers sont signalés dans une section ou un fichier dédié.
- [ ] Les fichiers sont organisés dans des dossiers compréhensibles.
- [ ] Les fichiers importants ont des noms simples, sans espaces ni accents.
- [ ] Les formats sources modifiables sont fournis quand c’est pertinent.
- [ ] Les liens internes fonctionnent.
- [ ] Les images s’affichent correctement.
- [ ] GitHub Pages est activé si un site public est prévu.
- [ ] L’URL publique a été testée.
- [ ] Le dépôt ne contient pas de données personnelles ou confidentielles.

Checklist accessibilité minimale

## Checklist accessibilité minimale

- [ ] La page contient un titre principal clair.
- [ ] Les titres sont hiérarchisés correctement.
- [ ] Les paragraphes sont courts et lisibles.
- [ ] Les liens ont des intitulés explicites.
- [ ] Les images importantes ont un texte alternatif.
- [ ] Les informations importantes ne sont pas uniquement dans des images.
- [ ] La couleur n’est pas le seul moyen de transmettre une information.
- [ ] Les contrastes sont suffisants pour une lecture confortable.
- [ ] Les tableaux sont simples et disposent d’en-têtes compréhensibles.
- [ ] Les fichiers lourds sont évités ou signalés.
- [ ] Les documents sont disponibles dans un format modifiable quand c’est pertinent.
- [ ] Les consignes sont compréhensibles hors contexte.
- [ ] La ressource est testée sur mobile.
- [ ] Les liens et fichiers téléchargeables fonctionnent.

À retenir

Les modèles ne remplacent pas la réflexion sur les droits, les usages et les publics. Ils servent à éviter la page blanche, homogénéiser les dépôts et rendre les ressources plus faciles à reprendre. Le bon modèle est celui que l’on adapte proprement.
